Lab 5 · Automated Composition

Pitch Set Composer

Spring 2026  ·  COMS 3430  ·  Pitch Set Theory (Part I selection)

In pitch set theory, the compositional grammar is the Dihedral-12 group, and every operation you apply is simultaneously a group element and a musical gesture.

Coding this compositional grammar helped me visualize the relationship between mathematical operations and musical structures.

What I Built

This pitch set composer lets you enter an initial pitch class sequence, select which operations from the D₁₂ group you want to allow, and generates a multi-phrase composition by randomly applying those operations in sequence. It plays back the result via the Web Audio API, with a pitch clock visualization and piano roll to show you what's happening as it plays.

The core 12-tone theory operations are shown below:

OperationFormulaGroup StructureMusical Effect
Tₙ (p + n) mod 12 Rotation in D₁₂ Shifts all pitches up by n semitones
I (12 − p) mod 12 Reflection in D₁₂ Mirrors the interval structure
TₙI (n − p) mod 12 Composition of T and I Inversion transposed to new starting pitch
R reverse ordering Temporal (not D₁₂ strictly) Plays the sequence backwards
RI reverse ∘ invert Combination Inverted and reversed

The composition generator picks from this pool randomly at each step, building a chain of phrases. The Web Audio engine then schedules these as actual sound using AudioContext.currentTime for sample-accurate timing.

Significance of the Group Structure

When I first wrote the inverse function, I wrote p => 12 - p without the mod. For pitch class 0, this gives 12, which is outside Z₁₂. The fix is to add % 12 but the bug forced me to understand why the modular arithmetic is non-negotiable. It is because the group's closure property means every operation must map pitch classes back to pitch classes.

When I fixed this bug I found that the group structure provides a robust framework for generating coherent musical sequences.

Pitch Classes Are Abstract

Pitch set theory is octave-agnostic. Pitch class 0 is "C" — but it doesn't say which C. Pitch class 7 is "G" — but which G? When you play back a generated composition, you have to make a choice the theory doesn't make for you.

After some experimentation, I ended up assigning a whole-phrase octave contour where each phrase gets a base octave, which shifts gently up and back over the course of the composition. This creates a sense of architectural rise and fall that sounds quite well.

The automated part of automated composition isn't doing as much work as it seems. The serial operations are generative, but the octave mapping is a layer of algorithmic-design choice that shapes the perceptual result.

Retrograde Is Mathematically Symmetric but Perceptually Asymmetric

Consider the phrase [C, D, E, F] and its retrograde [F, E, D, C]. They have identical pitch content. In set-theoretic terms, they're the same set. But when played in sequence you hear two completely different gestures — one ascending, one descending. One feels like a question, the other like an answer (or vice versa, depending on the surrounding context).

Some of the structure that matters most for musical perception (narrative arc, temporal directionality, phrase-level tension and release) isn't captured by the pitch-class operations at all. The implementation in this project helped me understand this distinction.

Randomness as Compositional Grammar

Below is the random operation selection in my generator:

// Pick a random operation from the enabled pool
const op = pool[Math.floor(Math.random() * pool.length)];
current = op.fn(current);

The content of pool is what the composer controls. If you only enable T₁ through T₃, you get a composition that drifts chromatically upward in small steps. If you enable only R and I, you get constant pitch-space mirroring. If you include TₙI operations, the melodic contour becomes more jagged and unpredictable.

In this generator you're choosing the space of transformations the composition can make. The random walk happens inside the grammar you choose.

Scheduling Notes in WebAudio

One thing this lab reinforced about WebAudio specifically: the difference between scheduling audio and triggering audio is significant. One might set up an oscillator and start it when a setTimeout fires. This produces audible timing drift at any tempo above ~80 BPM because setTimeout is not sample-accurate.

A better approach is to pre-schedule all notes using AudioContext.currentTime as an absolute timeline, then use setTimeout for UI updates — highlighting which note is playing, updating the pitch clock visualization. In this way, the audio engine runs ahead of the UI callbacks.

// Schedule audio: sample-accurate via AudioContext clock
osc.start(t);  osc.stop(t + duration);

// Schedule UI: approximate, via wall-clock delay
setTimeout(() => highlightNote(phraseIdx, noteIdx), wallDelay);
I

Implementing this separation made me appreciate the Web Audio API design. The AudioContext clock is isolated from the JavaScript event loop so that audio scheduling can be deterministic even when the UI thread is busy.

Notes for Next Time

The most obvious limitation of my implementation is that it operates on pitch classes as an undifferentiated sequence so that every note gets the same duration. Extending the system to serialize note durations would be a natural next step and would produce more varied output.

Conclusion

Implementing pitch set theory taught me about a deeper level of pitch-class relationships in music.

But music isn't just pitch-class relationships. It's time, timbre, register, dynamics, and the way a listener's working memory stitches phrases together into narrative. The pitch set operations handle one dimension of that, but the more complex parts would be decided by the composer.

That boundary between the formal and the expressive is what makes this an interesting domain. Not because the math is sophisticated, but because implementing it reveals the limitations of this formal approach.
